# Проект BSA15_REST — Проектирование и документирование интеграционных потоков (REST API)

Учебный проект специализированного трека «Бизнес- и системный анализ» в образовательной экосистеме «Школа 21» от Сбера. 

---

## 🎯 Описание и цели проекта
Проект посвящен глубокому изучению принципов межсистемного взаимодействия современных ИТ-архитектур в стиле **REST (Representational State Transfer)**. Главная цель — на практических кейсах пройти полный цикл сквозного проектирования интеграции между независимыми информационными системами, от сбора терминологии и актуализации пользовательских сценариев до детального маппинга данных и документирования кодов ошибок.

---

## 🛠 Технологии и инструменты
В ходе выполнения проекта использовался профессиональный ИТ-стек системного анализа:
*   **Архитектурный стиль:** REST API (протокол передачи данных HTTP/HTTPS).
*   **Форматы обмена данными:** JSON (JavaScript Object Notation), XML (eXtensible Markup Language).
*   **Моделирование и визуализация:** Нотация UML (диаграммы прецедентов Use Case, диаграммы последовательности Sequence Diagram).
*   **Инструменты документирования:** Swagger / OpenAPI спецификации, PlantUML, системы управления базами знаний (Confluence), таблицы маппинга данных.

---

## 📝 Структура выполненных задач (Содержание проекта)

Проект включает в себя 10 последовательных практических упражнений, закрывающих ключевые хард-скиллы системного аналитика:

1.  **Упражнение 00. Терминология** — Формирование единого глоссария интеграционного проекта. Синхронизация понятийного аппарата ИТ-систем.
2.  **Упражнение 01. Подготовка данных** — Анализ исходных сущностей данных, определение форматов, типов данных и обязательности полей для передачи.
3.  **Упражнение 02. Актуализация Use Case** — Проектирование и обновление пользовательских сценариев с учетом появления интеграционного слоя.
4.  **Упражнение 03. Интеграция справочников** — Описание механизмов синхронизации мастер-данных и нормативно-справочной информации между системами.
5.  **Упражнение 04. Определение и описание методов** — Проектирование архитектуры эндпоинтов, выбор HTTP-методов (GET, POST, PUT, DELETE) для реализации бизнес-логики.
6.  **Упражнение 05. Описание элементов запроса/ответа** — Детальная спецификация структур заголовков (Headers), тела запроса (Request Body) и ответов (Response Body).
7.  **Упражнение 06. Картирование (Маппинг данных)** — Составление сквозных таблиц сопоставления полей системы-источника и системы-приемника, описание правил трансформации данных.
8.  **Упражнение 07. Описание ошибок** — Проектирование матрицы обработки исключительных ситуаций. Документирование стандартных HTTP-статусов (4xx, 5xx) и внутренних бизнес-кодов ошибок систем.
9.  **Упражнение 08 (Звездное). Актуализация Use Case** — Расширенное моделирование сложных и альтернативных сценариев взаимодействия систем при пиковых нагрузках.
10. **Упражнение 09 (Звездное). Определение и описание методов** — Проектирование дополнительных и оптимизированных методов API для специфических b2b-сценариев интеграции.

## 🎓 Чему я научился в этом проекте (Ключевые навыки)
*   **Проектировать интеграции с нуля:** Переводить абстрактные требования бизнеса в строгие технические контракты взаимодействия систем (API Contracts).
*   **Работать с форматами данных:** Читать, понимать и самостоятельно составлять валидные сообщения в форматах JSON и XML.
*   **Связывать бизнес и разработку:** Описывать логику интеграции через Use Case и UML-диаграммы так, чтобы у разработчиков бэкенда не оставалось двусмысленностей при написании кода.
*   **Управлять качеством данных:** Проводить маппинг (картирование) данных любой сложности, предусматривая правила валидации и трансформации форматов «на лету».
*   **Проектировать отказоустойчивость:** Заранее закладывать в архитектуру понятные контракты ошибок, что значительно ускоряет последующее тестирование (QA) и поддержку продукта.
